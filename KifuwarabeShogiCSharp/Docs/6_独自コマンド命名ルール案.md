# 独自コマンド命名ルール案

このメモは、`KifuwarabeShogiCSharp` で USI には存在しない**独自コマンド**を追加するときの命名ルール案です。  
学習用プロジェクトなので、**短さより意味の分かりやすさ**を優先します。


## 結論

独自コマンドは、次の方針で付けるのが分かりやすいです。

- 動詞から始める
- 何を対象にするかを後ろに置く
- 省略しすぎない
- USI 標準コマンドと紛らわしい名前を避ける

例えば、局面表示なら `pos` より `show board` の方が分かりやすいです。


## なぜ `pos` より `show board` がよいか

`pos` は短いですが、少し曖昧です。

- `position` の略に見える
- USI にはすでに `position` コマンドがある
- 「局面を表示する」のか「内部状態を出す」のかが分かりにくい

一方、`show board` なら次が明確です。

- `show` = 表示したい
- `board` = 盤面を見たい

読みやすく、あとでコマンドを増やすときにもそろえやすいです。


## 基本ルール

### 1. 動詞 + 対象 にする

おすすめの形:

- `show board`
- `show hands`
- `show position`
- `dump sfen`
- `clear screen`

この形なら、見ただけで意味が追いやすいです。


### 2. 省略形はできるだけ避ける

避けたい例:

- `pos`
- `brd`
- `sh`

短くても、あとで読んだとき説明しにくいです。


### 3. USI 標準コマンドと衝突しないようにする

例えば次は、USI 側と混ざりやすいです。

- `position`
- `go`
- `stop`
- `ponder`

独自機能なら、標準コマンドとは別の見た目にした方が安全です。


### 4. 画面表示系は `show` に寄せる

画面に何か見せたいだけなら、まず `show` を使うと統一しやすいです。

例:

- `show board`
- `show hands`
- `show help`
- `show config`


### 5. 内部状態の吐き出しは `dump` に寄せる

人間向けの見やすい表示ではなく、調査用に生データを出すなら `dump` が合います。

例:

- `dump sfen`
- `dump position`
- `dump log-settings`


## このプロジェクト向けのおすすめ

現時点では、次のような住み分けが分かりやすいです。

### 表示系

- `show board`
- `show hands`
- `show help`

### 調査系

- `dump sfen`
- `dump position`

### 操作系

- `clear`
- `reset`


## 最小ルール案

迷ったら、ひとまず次の 3 つだけ守れば十分です。

1. **動詞 + 対象** にする
2. **USI 標準コマンドと似せない**
3. **略しすぎない**


## 今回の判断

今回の `pos` は、次の名前に変えるのがよいです。

- 変更前: `pos`
- 変更後: `show board`

理由:

- 短すぎず意味が分かる
- `position` と混同しにくい
- 将来 `show hands` や `show help` を増やしやすい
