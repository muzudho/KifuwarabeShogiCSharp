# フォルダー構成案

このメモは、`KifuwarabeShogiCSharp` を**学習しやすく、あとで読み返しやすい構成**に寄せるための案です。  
厳密な正解を決めるというより、**責務が見えること**を重視します。


## 結論

今の `Core` は間違いではありません。  
ただし、将棋そのものの概念と、USI という外部プロトコルの概念が混ざって見えやすいです。

学習用として読みやすくするなら、次の考え方が分かりやすいです。

- 将棋そのものの知識は `Domain`
- アプリの操作や処理の流れは `Application`
- 画面表示は `Presentation`
- 設定やログなど外部都合は `Infrastructure`
- USI のような通信規約は `Protocols`


## おすすめ構成

メインプロジェクトの目標構成は、例えば次のような形です。

```text
KifuwarabeShogiCSharp/

│     ├─ Board/
```

`Program.cs` は入口なので、ルート直下のままで問題ありません。


## 今の構成をどう見るか

今あるフォルダーは、大枠では次のように理解できます。

- `Core`
    - 中心ロジック置き場という意味では自然
    - ただし意味が少し広い
- `Infrastructure`
    - 設定、ロギングが入っていて分かりやすい
- `Views`
    - 表示責務が明確
- `Models`
    - 何のためのモデルかが少し分かりにくい

特に気になるのは、`Core\Usi\...` の中に、次の 2 種類が同居している点です。

- 将棋そのものの概念
    - 盤
    - 局面
    - 駒
    - 筋、段、升
- USI という規約に依存する概念
    - コマンド
    - SFEN 文字列
    - USI 入出力

将来的には、この 2 つは分けた方が理解しやすいです。


## このプロジェクト向けの具体案

今あるファイルを、次のように読むと整理しやすいです。

### 1. `Domain`

将棋そのものの知識です。  
「USI を知らなくても成立するもの」は、まずここを疑います。

移動候補の例:

- `Core\Usi\Elements\MuzSujiType.cs`
- `Core\Usi\Elements\MuzDanType.cs`
- `Core\Usi\Elements\MuzColorType.cs`
- `Core\Usi\Models\Position\MuzPositionModel.cs`
- `Core\Usi\Models\Position\Elements\MuzBoardModel.cs`
- `Core\Usi\Models\Position\Elements\MuzMasuType.cs`
- `Core\Usi\Models\Position\Elements\MuzKomaType.cs`
- `Core\Usi\Models\Position\Elements\MuzHandStandCollectionModel.cs`

例えば、次のような寄せ方が考えられます。

```text
Domain/
└─ Shogi/
   ├─ Coordinates/
   │  ├─ MuzSujiType.cs
   │  ├─ MuzDanType.cs
   │  ├─ MuzMasuType.cs
   │  ├─ MuzSujiHelper.cs
   │  └─ MuzDanHelper.cs
   ├─ Turns/
   │  ├─ MuzColorType.cs
   │  └─ MuzColorModel.cs
   ├─ Position/
   │  ├─ MuzPositionModel.cs
   │  └─ MuzPositionModelReadonly.cs
   ├─ Board/
   │  ├─ MuzBoardModel.cs
   │  └─ MuzBoardModelReadonly.cs
   ├─ Hands/
   │  ├─ MuzHandStandCollectionModel.cs
   │  └─ MuzHandStandCollectionModelReadonly.cs
   └─ Pieces/
      └─ MuzKomaType.cs
```


### 2. `Protocols\Usi`

USI という規約に関するものです。  
文字列の解釈、出力形式、USI コマンド処理がここに入ります。

現時点では、`ProgramCommands.cs` に USI コマンドが混ざっています。  
学習用に読みやすくするなら、いずれ次のように分けると見通しが良いです。

```text
Protocols/
└─ Usi/
   ├─ Commands/
   │  ├─ UsiCommandHandler.cs
   │  └─ PositionCommandHandler.cs
   ├─ Parsing/
   └─ Formatting/
```

ただし、今すぐ大きく分割しなくても構いません。


### 3. `Application`

アプリとして何をするかを表す層です。  
「将棋そのもの」ではなく、「このアプリでの処理の流れ」を置きます。

候補:

- `ApplicationService.cs`
- `ProgramCommands.cs`

`ProgramCommands.cs` は、現状では次が混ざっています。

- REPL コマンド受付
- USI コマンド受付
- 独自コマンド受付
- 表示呼び出し

学習用としては、最初はここにまとまっていても読めます。  
ただし育ってきたら、`Application\Commands` へ切ると責務が見えやすくなります。


### 4. `Presentation`

コンソール表示など、見せ方の責務です。

候補:

- `Views\MuzPositionView.cs`
- `Views\MuzKomaTypeView.cs`
- `Models\MuzCoreModelReadonly.cs`

特に `MuzCoreModelReadonly` は、名前だけでは役割が伝わりにくいです。  
表示用の読み取りモデルなら、次のような置き場の方が読みやすいです。

- `Presentation\ViewModels`
- `Presentation\Models`


### 5. `Infrastructure`

ここは今のままで分かりやすいです。

- `Infrastructure\Configuration`
- `Infrastructure\Logging`

この方向で問題ありません。


## 命名の考え方

学習用プロジェクトでは、**短い名前より、役割が伝わる名前**の方が有利です。

### `Core` の良い点

- 短い
- 中心にあるコードだと分かる
- 小規模では使いやすい

### `Core` の弱い点

- 何でも入りやすい
- 将棋ルールと USI と表示補助が混ざっても違和感が出にくい
- 後で読むと、なぜそこにあるか説明しにくい

### `Domain` の良い点

- 将棋そのものの知識置き場だと分かる
- 学習時に「これは業務知識か、道具か」を分けやすい
- 将来の分割に強い

そのため、このプロジェクトでは **`Core` より `Domain` の方が教材として説明しやすい** です。


## まずはこう直すのがおすすめ

一気に大改造すると、学習用としてはかえって追いにくくなります。  
次の順で少しずつ寄せるのが安全です。

### 段階 1

- `Core` を `Domain` に置き換える方針を決める
- ただし、すぐには全ファイルを動かさない
- ドキュメントだけ先に整える

### 段階 2

- `Core\Usi\Models\Position` 配下を、将棋モデルとして `Domain` 側へ寄せる
- `Suji`、`Dan`、`Masu`、`Koma` のような基礎要素も `Domain` へ寄せる

### 段階 3

- USI に依存するコードだけを `Protocols\Usi` に分離する
- `position`、`go`、`usi` などのコマンド処理を段階的に分ける

### 段階 4

- 表示用モデルを `Presentation` に寄せる
- `Views` を `Presentation\Views` に寄せるか、`Views` のままにするか決める


## 学習用としてのおすすめ判断

学習目的を重視するなら、次の方針が特におすすめです。

1. **フォルダー名を見れば責務が分かること**
2. **同じ概念を近くに置くこと**
3. **外部規約と将棋そのものを分けること**
4. **一度に全部変えず、段階的に変えること**

この観点なら、現時点での最適解は次の一言にまとまります。

> 将棋コードは `Core` より `Domain` に寄せ、USI は `Protocols\Usi` として分けると読みやすい。


## 最終メモ

今すぐ必要なのは、完璧な階層設計よりも、**あとで見たとき説明できる構成**です。  
その意味では、次の形がいちばん扱いやすいです。

- `Domain` = 将棋そのもの
- `Application` = このアプリの処理の流れ
- `Presentation` = 表示
- `Infrastructure` = 設定、ログ、外部都合
- `Protocols` = USI のような通信規約

このメモを土台に、実際のリファクタリングは小さく進めるのがよいです。
